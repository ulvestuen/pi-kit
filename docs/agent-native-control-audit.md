# Agent-native control audit: planner, critic, and lykkja

**Audit baseline:** `feature/agent-native-orchestration` at `a84eb7c` (2026-07-11). This is a focused audit of the current planner/critic/lykkja control contracts and the orchestrator paths that compose them. It records current behavior and constraints; it does not select or implement the replacement architecture.

## 1. Current control and data contracts

### Planner: plan data, DAG, and status

- `Plan` is one goal, an ordered task array, and create/update timestamps. Each `PlanTask` stores `id`, title, a free-text self-contained description, dependency ids, optional agent name, normalized criteria, status, and dispatch-attempt count (`planner/plan.ts:37-73`). There is no run id, task-output/artifact field, review/evidence record, status history, criterion version, or owner/actor identity.
- Criteria are the shared lykkja `Criterion { name, threshold }` type, imported with `normalizeCriteria` from the pure loop module (`planner/plan.ts:10-16`; `lykkja/loop.ts:13-31`). Creation trims required fields, normalizes criteria, initializes every task as `pending` with zero attempts, and leaves `agent` absent when omitted (`planner/plan.ts:83-113`). Thus `PlannerConfig.defaultAgent` is advertised and loaded (`planner/config.ts:13-23,115-122`) but is not applied by planner; the actual orchestration fallback is separately configured at dispatch (`orchestrator/index.ts:291-296`).
- DAG validation is case-insensitive for ids and rejects duplicate ids, self/dangling dependencies, and cycles (`planner/plan.ts:116-169`). `readySet` mechanically admits only `pending|ready` tasks whose dependencies are all `done` (`planner/plan.ts:202-208`). Summaries preserve status counts, transitive failure blockers, unfinished critical-path length, and all-done detection (`planner/plan.ts:304-385`).
- The status vocabulary is `pending|ready|running|review|done|failed` (`planner/plan.ts:20-35`), but `setTaskStatus` validates only membership, not legal transitions or dependency/review preconditions; any caller can move any known task directly to any status. Entering `running` from another status increments attempts (`planner/plan.ts:211-240`). The planner tool relies on prompt guidance—not an enforced state machine—to reserve `done` for verified work (`planner/index.ts:170-181,213-250`). The orchestrator's pure scheduler supplies the stricter normal path: dispatch -> `running`, successful execution -> `review`, passed review -> `done`, and failures -> `ready|failed` at the attempt cap (`orchestrator/scheduler.ts:80-117,119-159`).
- Task edits can replace title, brief, agent, and criteria at any status; appended tasks trigger full DAG revalidation, but changing existing dependencies is unsupported (`planner/plan.ts:243-301`). There is no immutable original brief/rubric or record that a completed review scored the same criterion version now visible.

### Critic: prompt, result, and evidence boundary

- `ReviewRequest` is free-text `subject`, optional free-text `context`, criteria, and `scaleMax`; `ReviewResult` is lykkja-shaped scores, aggregate pass, prioritized weaknesses, and raw critic prose (`critic/review.ts:19-45`). The prompt includes subject/context and exact criterion names/thresholds, orders the critic to inspect files independently, and requires one fenced JSON score block (`critic/review.ts:47-95`). Advice is deliberately unscored (`critic/review.ts:97-116`).
- Output parsing prefers the last fenced block, falls back to a bare object, canonicalizes criterion names case-insensitively, clamps/rounds scores, requires all requested criteria, ignores extras, and orders failures by threshold margin (`critic/review.ts:118-227`). No parse or missing criterion is a loud failed review with `scores: []`; a below-threshold score without a weakness gets a synthesized weakness only in the aggregate list (`critic/review.ts:122-129,194-225`). Duplicate output names are not rejected—the last map entry wins (`critic/review.ts:179-191`).
- The extension discovers a configured `critic` agent or uses a built-in fallback restricted to `read|grep|find|ls`, dispatches one fresh child through fleet/spawn, and retries an unscorable response once (`critic/index.ts:25-40,82-115,157-193`). The shipped critic is likewise read-only (`fleet/agents/critic.md:1-24`). This protects the tree and context independence, but it cannot execute tests, type-checks, benchmarks, or runtime probes; runtime evidence must already exist in inspectable files or be trusted from the implementer report.
- The standalone tool returns the rendered verdict plus full `ReviewResult` in tool `details` (`critic/index.ts:185-193`). The critic extension has no custom review-state entry: its only `session_start` action cleans stale spawn jobs (`critic/index.ts:117-119`). Although ordinary tool messages may remain in the parent transcript, there is no queryable, versioned critic evidence store. `dispatchCritic` also narrows the fleet result to only output/status, dropping the saved transcript path and other task metadata (`critic/index.ts:94-115`; compare `fleet/runner.ts:34-47,100-108`).

### Lykkja: criteria, checkpoint, and ACT

- `LoopState` owns one task, normalized criteria, scale/default threshold, iteration cap, all iterations, status, and timestamps. Each iteration persists the PLAN string, DO summary, every score, pass bit, weakest name, and timestamp; decisions are `FINAL|ITERATING|STOPPED` with failing scores (`lykkja/loop.ts:33-99`).
- Criterion normalization rejects empty/duplicate names and out-of-range thresholds, while score reconciliation clamps values and requires exactly one known score for every criterion (`lykkja/loop.ts:109-161,202-252`). Despite comments and tool descriptions saying a weakness is required below threshold, core reconciliation does not enforce it (`lykkja/loop.ts:24-31,224-228`; `lykkja/index.ts:297-313`).
- `recordCheckpoint` accepts only an active loop and nonempty PLAN/DO summaries, appends one complete iteration, ranks failures by threshold margin, and changes status to `final` when all pass or `stopped` at the cap (`lykkja/loop.ts:259-335`). The ACT text identifies the weakest criterion and the extension returns the next model instruction (`lykkja/loop.ts:337-369`; `lykkja/index.ts:85-130,316-333`). The state machine keeps score; the parent model still performs the plan, work, evidence collection, and follow-up tool call (`lykkja/README.md:252-267`).
- One extension instance holds one `LoopState`; `lykkja_start` replaces it without a run/link id (`lykkja/index.ts:202-210,255-273`). This is compatible with one goal loop per parent session but cannot natively represent a goal loop plus multiple task loops in one session.

## 2. Composition and handoffs

1. The model creates goal criteria in `lykkja_start`, then separately creates task criteria in `plan_create`; there is no structural relation from goal criteria to task criteria (`orchestrator/index.ts:60-75`).
2. Planner criteria reach implementers as rendered prompt text alongside the goal and task brief (`orchestrator/index.ts:78-95`). Dependency outputs are not passed as typed inputs; children infer them from the shared/worktree filesystem and the description.
3. Successful child output moves a task to `review`. Review context consists of task id/title, the capped final implementer report, current-tree/branch wording, the goal, and the current task description; critic cwd is switched to the task worktree when present (`orchestrator/index.ts:98-114,314-347`). `TaskResult.fullOutputPath`, prior dependency results, changed-file metadata, commands run, and structured artifacts are not included.
4. A parsed passing task review changes only status to `done`. A failed execution or review is persisted by appending unstructured failure/weakness text into `task.description`, then re-queuing or failing the task (`orchestrator/scheduler.ts:90-117,124-159`). Consequently failed feedback survives in the plan, but successful scores, raw review, evidence provenance, and the implementer result do not.
5. Each wave returns structured `details.results`, `details.reviews`, and a plan summary (`orchestrator/index.ts:474-483`), then asks the parent model to translate task-level critic verdicts into scores for every **goal-level** criterion and call `lykkja_checkpoint` (`orchestrator/index.ts:460-471`). This is a prompt-mediated semantic handoff, not a typed function or enforced evidence mapping. The existing design explicitly describes the critic-to-checkpoint helper as optional future work (`docs/multi-agent-orchestration.md:480-491`).
6. A successful `critic_review.scores` array is shape-compatible with a lykkja checkpoint. An unscorable review's empty scores are not: lykkja rejects missing scores, so the caller must retry or synthesize a complete failing checkpoint rather than literally feed the result through (`critic/review.ts:122-129`; `lykkja/loop.ts:240-245`). `raw`, prioritized weaknesses, and evidence location are discarded by the checkpoint shape.

## 3. Persistence, events, and restart behavior

### Session entries

- Planner appends whole-plan snapshots under `planner-state` after create/update/clear and restores the latest entry (`planner/index.ts:110-130,152-165,247-255,302-308`). On session start, all restored `running` tasks reset to `ready` so lost children can be re-dispatched (`planner/index.ts:277-291`). Restoration validates only that `goal` is a string—there is no schema version, migration, full DAG validation, optimistic version, or corrupt-entry diagnostic.
- Lykkja appends whole-loop snapshots under `lykkja-state` on start/checkpoint and scans for the latest object with a string task (`lykkja/index.ts:17-18,41-53,209-210,255-273,321-348`). `/lykkja reset` appends an `undefined` tombstone (`lykkja/index.ts:386-390`), but `restoreState` ignores non-object entries rather than setting `latest = null`; an older loop can therefore reappear after reload. Planner's restore explicitly interprets its tombstone as null (`planner/index.ts:121-129`).
- Orchestrator independently scans planner's entry type on every wave and persists only `{stopped,wave}` under `orchestrator-state` (`orchestrator/index.ts:44-58,149-165,202-207,489-499`). There is no shared orchestration/run id linking orchestrator state, plan snapshots, lykkja checkpoints, fleet jobs, and reviews. Concurrent or replaced runs in one session are therefore correlated by “latest entry” and convention.
- Spawned fleet children use `pi --mode json --no-session` (`fleet/runner.ts:118-134`). They have isolated context/processes but no durable child session. This qualifies the design claim that task-level loops have their own resumable entry logs (`docs/multi-agent-orchestration.md:472-478`): a child can loop during its single process, but its lykkja state does not survive as a child session.

### Shared event bus

- Planner emits `planner:plan_created`, `planner:plan_updated`, and per-requested-status `planner:task_status`; it also accepts `planner:set_plan` (`planner/index.ts:152-165,223-250,261-268`). The orchestrator publishes whole replacement plans through only that event (`orchestrator/index.ts:167-170,263-267,306-312,389-396`).
- `planner:set_plan` checks only for a string goal, then replaces and persists the plan; it does not revalidate the DAG/statuses, update the planner status bar, emit `plan_updated`, acknowledge the write, or carry source/run/version metadata (`planner/index.ts:261-268`). The bus payload is untyped and last-writer-wins. Status events can also be emitted before the complete `plan_update` transaction is persisted, so a later edit/add failure can leave observers with an announced but uncommitted status (`planner/index.ts:213-250`).
- Orchestrator reads planner state back through the session log rather than a planner query API (`orchestrator/index.ts:155-170`). Correctness therefore depends on synchronous event handling/session-entry visibility and extension load, with no acknowledgement or compare-and-swap. Wave start/end and fleet progress events are observational only and contain no durable correlation beyond wave/index (`orchestrator/index.ts:254-261,297-305,399`).
- Planner and lykkja inject independent prompt protocols on `before_agent_start` (`planner/index.ts:270-275`; `lykkja/index.ts:338-343`). Their operational composition order is extension-load order plus model compliance, not a unified controller contract.

## 4. Barriers to structured native agent development

These are the concrete gaps between the current prompt-driven composition and a structured, inspectable native-agent control plane:

1. **No stable run/task/attempt/review identity.** Attempts are counters, waves are counters, and latest session snapshots are correlated by timing. Results, reviews, checkpoints, events, and branches cannot be joined reliably after restart or across concurrent runs.
2. **Evidence is not first-class.** Plan tasks have no outputs/artifacts/evidence/reviews; passed review evidence disappears from persisted plan state; lykkja scores carry no source, command, artifact hash, reviewer, or criterion version. The strongest evidence currently lives transiently in tool `details`, child temp transcripts, and prose.
3. **Context assembly is free text.** Task and critic inputs are rendered strings. Dependency results, permitted scope, changed files, verification commands/results, and artifact references have no schema, size policy, or provenance. Fresh context reduces bias but also drops unstated constraints.
4. **Control depends on the parent model obeying prompts.** The model must open the right loop, create the plan, call waves/checkpoints, derive goal scores, repair blocked plans, and stop on verdict. Planner permits arbitrary status transitions, and no controller atomically gates `done` on a matching passing review.
5. **Task and goal quality gates are disconnected.** The same criterion shape is reused, but there is no typed mapping/aggregation rule from task reviews to goal criteria. “Critic-derived goal scores” are parent-model judgments, not critic results enforced by code.
6. **Independent scales can be incompatible.** Planner, critic, and lykkja each configure their own scale/pass defaults (`planner/config.ts:96-122`; `critic/config.ts:101-135`; `lykkja/config.ts:95-130`), while `Plan` stores thresholds but not its scale. Orchestrator hard-codes lykkja's default scale 10 for task briefs/reviews (`orchestrator/index.ts:275-276,332-340`). A plan normalized on another scale can become impossible or semantically distorted; a standalone critic result can likewise be shape-compatible but scale-incompatible with the active loop.
7. **Persistence is snapshot/tombstone convention, not an event model.** Whole mutable objects are appended without schema/version validation, causality, ownership, idempotency keys, or conflict handling. Event-bus replacement and session-log reads create dual state-transfer paths.
8. **A read-only critic cannot independently run executable checks.** It can inspect source and test definitions but not reproduce “tests pass,” performance, generated output, or environment behavior. The contract currently conflates repository inspection with complete evidence verification.
9. **Child durability and nested loops are limited.** `--no-session` gives clean context but prevents resumable task-level sessions and persisted child PDCA history. Parent restart recovery resets work rather than reattaching to a structured child attempt.
10. **Configuration and packaging are only aggregate-safe.** Cross-package source imports and tsconfigs require sibling trees (`planner/tsconfig.json:17`; `critic/tsconfig.json:17`; `orchestrator/tsconfig.json:17-23`), while package manifests declare no runtime workspace dependencies (`planner/package.json:1-19`; `critic/package.json:1-19`; `orchestrator/package.json:1-19`). The READMEs correctly require sibling folders/vendor copies for standalone installs (`planner/README.md:60-62`; `critic/README.md:66-70`; `orchestrator/README.md:93-95`). Root installation supplies all extensions/skills together (`package.json:9-41`).

## 5. Invariants to preserve in an agent-native replacement

- **Validated acyclic work graph:** unique case-insensitive ids, no dangling/self edges/cycles, dependency-gated ready set, explicit failure blockage (`planner/plan.ts:116-169,202-208,318-345`).
- **Shared strict rubric semantics:** unique named criteria with explicit inclusive thresholds; exact one-score-per-requested-criterion reconciliation; score clamping; all criteria must pass—never average away a failure (`lykkja/loop.ts:109-161,202-252,295-322`; `critic/review.ts:194-225`).
- **Independent gate and fail-closed parsing:** implementation reports remain informational; malformed/partial critic output cannot pass, and critic failure is actionable/retryable (`critic/review.ts:151-227`; `orchestrator/index.ts:379-395`).
- **Deterministic hard control:** bounded concurrency, explicit `running -> review -> done`, bounded retries, blocked/complete terminals, and a finite PDCA safety cap (`orchestrator/scheduler.ts:53-77,80-159`; `lykkja/loop.ts:313-322`).
- **Context isolation and least privilege:** children receive a self-contained scoped brief; critics get fresh context and cannot modify work; optional worktrees isolate parallel writers (`planner/skills/plan-decomposition/SKILL.md:27-37`; `orchestrator/index.ts:78-95,332-347`).
- **Honest restart/stop semantics:** do not silently mark interrupted work complete; reset/requeue in-flight work, preserve visible failures, and report STOPPED rather than manufacture success (`planner/index.ts:277-290`; `lykkja/loop.ts:313-369`).
- **Standalone pure cores:** DAG/scoring/review/scheduler logic remains deterministic and independently testable; extension wiring should not be imported for composition (`planner/plan.ts:1-7`; `critic/review.ts:1-8`; `lykkja/loop.ts:1-7`; `orchestrator/scheduler.ts:1-8`).

Verification on the audit baseline: `npm test --workspace={planner,critic,lykkja,orchestrator}` passed 27/27, 17/17, 34/34, and 16/16 tests respectively; `npx tsc -p <package>/tsconfig.json` passed for all four packages. These results validate the pure contracts above, not the explicitly uncovered ExtensionAPI path.

Compatibility constraints for replacement work:

- Preserve current public tool names and primary parameter/result shapes unless a migration layer is supplied: `plan_create`, `plan_update` (`planner/index.ts:132-212`), `critic_review`, `critic_advise` (`critic/index.ts:121-235`), and `lykkja_start`, `lykkja_checkpoint` (`lykkja/index.ts:213-335`). Existing skills and orchestrator prompts call those names directly.
- Preserve the six task-status strings and `FINAL|ITERATING|STOPPED` verdict strings because tests, dashboards, prompts, scheduler, and persisted snapshots compare literals (`planner/plan.ts:20-35`; `lykkja/loop.ts:53,86`; `orchestrator/scheduler.ts:61-75`).
- Treat `planner-state`, `lykkja-state`, and `orchestrator-state` as persisted compatibility formats or provide explicit migrations (`planner/index.ts:22`; `lykkja/index.ts:17`; `orchestrator/index.ts:44-45`). Old entries currently receive only shallow validation.
- Keep config defaults and environment/file lookup behavior compatible or document migration. All three default to scale 10/pass 8, lykkja caps at 25 passes, and invalid configs fall back at extension startup (`planner/index.ts:92-107`; `critic/index.ts:62-79`; `lykkja/index.ts:184-200`).
- Preserve pi `^0.71.0`, ES module/TypeScript-extension imports, strict no-emit builds, and root aggregate discovery unless the larger architecture explicitly raises the compatibility floor (`planner/package.json:5-18`; `planner/tsconfig.json:2-17`; analogous critic/lykkja/orchestrator manifests).
- Before claiming end-to-end compatibility, add extension-level tests around session/event handoffs and run the existing package tests. The current documented manual smoke requires loading lykkja, fleet, planner, critic, and orchestrator together (`orchestrator/README.md:75-91`); pure tests alone do not verify that path.
